# invoice-web 개발 로드맵

관리자가 Notion에서 관리하는 인보이스를 클라이언트가 웹에서 직접 조회하고 PDF로 다운로드하는 인보이스 조회 시스템.

## 개요

invoice-web은 **인보이스를 발급받은 클라이언트(비로그인 외부 사용자)**를 위한 안전한 인보이스 조회 시스템으로 다음 기능을 제공합니다:

- **인보이스 조회 (F001/F002/F003)**: 인보이스 번호 + 접근 코드 입력 또는 공유 URL로 본인 인보이스에만 접근
- **인보이스 웹 표시 (F004)**: Notion 데이터를 전문 인보이스 양식(헤더/품목/합계/상태)으로 렌더링
- **PDF 다운로드 (F005)**: 브라우저 Print API(`window.print`)로 인보이스를 PDF로 저장
- **관리자 공유 기능 (F006)**: 대시보드에서 인보이스별 공유 URL과 접근 코드를 생성·복사

## 의존성 관계

```
Phase 1 (보안 기반) ──┬──> Phase 2 (조회) ──> Phase 3 (PDF)
   [기반 필수]        │       [상세 페이지]      [상세 페이지 위 동작]
                      │
                      └──> Phase 4 (대시보드 공유)
                              [관리자 인증 + 접근 코드 활용]

                                      ↓
                              Phase 5 (테스트 & 보안 검증)
                              [전 Phase 완료 후 통합 검증]
```

- **Phase 1**은 모든 후속 Phase의 전제 조건입니다 (인증·접근 코드·Notion 조회 기반).
- **Phase 3**은 Phase 2의 인보이스 상세 페이지가 완성되어야 시작 가능합니다.
- **Phase 4**는 Phase 1의 관리자 인증과 접근 코드 체계에 의존하지만, Phase 2/3와는 병렬 진행 가능합니다.
- **Phase 5**는 전 Phase의 기능이 구현된 후 통합 테스트로 진행합니다.

## 개발 워크플로우

1. **작업 계획**
   - 기존 코드베이스를 학습하고 현재 상태를 파악
   - 새로운 작업을 포함하도록 `ROADMAP.md` 업데이트
   - 우선순위 작업은 마지막 완료된 작업 다음에 삽입

2. **작업 생성**
   - `/tasks` 디렉토리에 새 작업 파일 생성 (명명: `XXX-description.md`)
   - 고수준 명세서, 관련 파일, 수락 기준, 구현 단계 포함
   - **API/비즈니스 로직 작업 시 `### 🧪 테스트 체크리스트 (Playwright MCP)` 섹션 필수** (정상/에러/보안 케이스 + 수락 기준 포함)
   - 초기 상태 샘플은 `000-sample.md` 참조 (빈 박스, 변경 요약 없음)

3. **작업 구현**
   - 작업 파일의 명세서를 따라 구현
   - **API 연동 및 비즈니스 로직 구현 시 Playwright MCP(`mcp__playwright__*`)로 E2E 테스트 수행 필수**
   - 각 단계 후 작업 파일 내 진행 상황 업데이트, 테스트 통과 후 다음 단계 진행
   - 각 단계 완료 후 중단하고 추가 지시를 대기

> **⚠️ 테스트 필수 규칙 (반드시 준수)**
> - API 연동/비즈니스 로직이 포함된 **모든 Task**는 구현 후 반드시 Playwright MCP(`mcp__playwright__*`)로 테스트한다
> - 각 Task에 `### 🧪 테스트 체크리스트 (Playwright MCP)` 섹션을 **필수** 작성한다
> - 테스트 시나리오는 **정상 케이스 + 에러/엣지 케이스 + (해당 시) 보안 케이스**를 모두 포함한다
> - 모든 테스트 항목이 통과해야만 Task를 완료(✅)로 표시할 수 있다 — 테스트 없는 완료는 **금지**된다

4. **로드맵 업데이트**
   - 완료된 작업을 ✅로 표시하고 진행률 테이블 갱신
